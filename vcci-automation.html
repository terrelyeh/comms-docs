VCCI 日本認證登錄自動化

送件 × 狀態同步 × 線上 Dashboard 的完整生命週期工具
建立 2026-04-19 · 更新 2026-04-19（加入狀態同步 + Dashboard）
TL;DR

從「20–30 分鐘送件」到「完整生命週期自動化」

EnGenius Networks Japan 每款新產品上市前都要在 VCCI 官網登錄。我們做了兩個半自動工具：vcci-submit（PDF → 自動填表 → 送出 → 歸檔）和 vcci-status-sync（每週抓 VCCI 官網狀態 → 回填 Acceptance No.），外加一個密碼保護的線上 Dashboard。

過程中碰到很多官方 CGI 網站的反自動化機制 — 合成事件被擋、AJAX 需要真實 blur、雙語 PDF 錯位。最終靠 Playwright 的 page.mouse.click()（產生 OS 級真實滑鼠事件）+ Cookie 持久化 profile 繞過所有障礙。

關鍵商業洞察：Report No. ≠ 可販售，要等 VCCI 4–5 天審核完發 Acceptance No. 才能合法出貨。所以「追蹤循環」和「送件」一樣重要。整套架構可複用到 FCC / CE / KC 等其他長週期認證送件、政府申報系統。

背景

為什麼要做這件事

EnGenius Networks Japan K.K. 在日本販售網通商品，每一款新產品上市前必須在 VCCI（電波障害自主規制協議会）官網登錄產品符合性報告。流程雖然不複雜，但：

  • 重複性高（每月 1–2 款）
  • 欄位細碎（13 個必填欄位 + AJAX 聯動）
  • 送錯難撤回（需要 VCCI 介入）
  • 送出去只是第一步：要等 4–5 個工作日核發 Acceptance No. 才能合法貼標販售

是典型「低頻、長週期、高單筆成本、易出錯」的工作，非常適合 AI + 半自動化處理。除了送件本身，還需要一個狀態追蹤循環來管理等待期。

痛點

自動化碰到的技術障礙

政府/法規類網站有很多反自動化機制，不像一般 SaaS 那樣 API-first。以下是我們實戰發現的主要障礙：

🔒 痛點 1：多層驗證（Basic Auth + Form Login）

官方網站常有兩層以上的驗證：第一層是瀏覽器級的 HTTP Basic Auth 彈窗，第二層才是網站內的 Email + Password 登入表單。Playwright 處理第一層靠 http_credentials 參數，但第二層才是真正難搞的地方。

🚫 痛點 2：合成事件不被信任（最大障礙）

VCCI 的 CGI 會靜默拒絕所有「非 OS 級」的 click 事件。以下四種方式測試結果：

Click 方式 事件類型 VCCI 反應
element.click()（JS 呼叫） 合成事件 靜默忽略
page.locator(...).click()（Playwright） 合成事件 靜默忽略
page.keyboard.press('Enter') 合成事件 不穩定
page.mouse.click(x, y)（OS 座標點擊） 真實事件 ✅ 接受

關鍵發現：page.mouse.click() 直接移動指針到座標、按下、放開，產生的是作業系統 input event，VCCI 會當成真人點擊。這是 Playwright 的隱藏武器。

🎯 痛點 3：AJAX 聯動需要真實 blur 事件

「Test Lab Member No.」欄位填完後應該 AJAX 自動帶出公司名。但 page.fill()（直接 set value）不會觸發 AJAX，因為沒發 blur。解法是模擬真實打字 + Tab：

Python / Playwright
field.click() field.press_sequentially(value, delay=30) field.press("Tab") # 觸發 blur → 啟動 AJAX page.wait_for_function("...CompanyName 有值")
📑 痛點 4：雙語 PDF 解析錯位

TÜV Rheinland 測試報告是德英雙語並列，兩個語言的 label 在文字流裡交替。用英文 label 當 regex anchor 會抓到下一個欄位的值（因為德文值先出現）。解法：改用德文 label 當 anchor，值就在同一行。

🔀 痛點 5：CGI 表單 — 一個 URL 多個狀態

VCCI 是老式 CGI 系統，所有頁面都是同一個 URL（Manager.pl），伺服器根據 POST 資料決定回哪個頁面。不能靠 URL 變化判斷進度，要 poll 下一個預期元素是否出現：

偵測頁面轉換
page.wait_for_selector( "input[type='submit'][value='Report']", timeout=30_000 )
🤖 痛點 6：Playwright 指紋被識別

VCCI 會檢測 Playwright 預設的 navigator.webdriver 等特徵拒絕登入。解法組合：真實 User-Agent + launch_persistent_context 真實 profile + 讓使用者手動登入一次，之後 cookie 持久化，繞過登入環節的 bot 檢測。

⏳ 痛點 7：非同步核發 — Report No. ≠ 可販售

這是商業面的痛點：VCCI 送件後不是立刻就能賣。

欄位 核發時機 意義
Report No. 送件當下 只代表「我送出去了」
Acceptance No. 4–5 個工作日後 真正能貼 VCCI 標、合法販售

如果沒做狀態追蹤，很容易「以為送了就能賣 → 實際上還在審核」。所以這類認證自動化專案，追蹤循環和送件本身一樣重要。

vcci-submit vcci-status-sync

同步流程

One command
uv run vcci-update --push

背後執行：

  1. 開 Chromium + 持久化 profile — cookie 還有效就直接進，否則手動登入一次
  2. 造訪 VCCI Top 頁（ME02），parse Recent reporting status 表格
  3. 存 snapshot 到 archive/status-snapshots/<timestamp>/（PNG + HTML + models.json）
  4. 比對 tracking.xlsx：pending 列若 portal 有 Acceptance No. 就回填、Status 改成 Accepted
  5. 重生 Dashboard，新狀態立刻反映到每個型號的 detail page
  6. Git commit + push → Vercel 自動重新部署
🎯
謹慎原則：status 只寫 terminal 狀態 Portal 的 in progress 和 tracking.xlsx 的 Submitted 是同義詞，不該互相覆蓋。同步邏輯只在真正變成 Accepted / Rejected 時才寫 Status cell，避免每週 sync 都製造 diff noise。
📸
Snapshot 累積，但頁面長度不變 每次同步都產生新的 snapshot 資料夾（完整 audit trail），但 dashboard 每個型號的 detail page 只顯示一張「最新一張含該型號」的 snapshot。還 pending 的型號永遠指向最新 snapshot；被 Accept 的型號會自動 freeze 在「消失前最後那張」。累積多週也不會讓頁面變長。
Dashboard

從 Excel 到線上分享

有了 tracking.xlsx 後，內部想知道狀態還得開 Excel — 對非開發同事、外部合作方都不友善。所以做了一個簡單的線上 Dashboard。

功能一覽

  • 首頁：所有送件的 rolling table、客戶端搜尋、統計 KPI
  • 每個型號的 detail page：狀態 banner、送件資訊、PDF 萃取出來用於填表的 YAML 資料、4 張送件截圖（04 自動更新到最新 sync）、原始 PDF + YAML 下載
  • 密碼保護：自訂 HTML 登入頁（純密碼、無 username 欄），cookie 記 30 天
  • 部署：Vercel 自動部署，每次 push 後 ~1 分鐘生效

生命週期視角：每個型號的「履歷表」

每個型號的 detail page 從送件當下開始累積：

時間點 detail page 看得到什麼
送件當天 完整表單截圖、確認頁、成功頁、送件當下的 Top 頁狀態
週期 sync 期間 04 截圖自動更新到最新 Top 頁 snapshot（in progress / Acceptance No. 已發）
Accept 後 04 截圖 freeze 在「Accept 前最後一張快照」，整頁變永久 audit

這是把「靜態送件紀錄」升級成「產品的認證履歷表」。

給其他自動化專案的 7 條教訓

Lesson 01

先確認「真人怎麼做」，再想「怎麼自動化」

剛開始以為 form login 可以直接 page.fill() + page.click()。搞了兩小時才發現 VCCI 根本不接受合成 click。一定要先用 Chrome DevTools 或 Claude in Chrome 手動走一次流程、錄下真實 HTTP request 與事件，再設計自動化，才能避開誤入歧途。

Lesson 02

page.mouse.click() 是 Playwright 的隱藏武器

大多數教學只講 locator.click()。但當遇到「合成事件被擋」的老舊 CGI 時，page.mouse.click(x, y) 才是真正產生 OS 級事件的方法。步驟是：scroll_into_view → bounding_box 算座標 → mouse.move → mouse.down → mouse.up。

Lesson 03

Cookie 持久化比「每次登入」更穩

launch_persistent_context + 固定 user_data_dir 讓 cookie 跨執行保留。好處：只要手動登入一次（繞過登入環節的 bot 偵測）→ 後續 run 全自動 → Profile 有真實瀏覽指紋，比 fresh context 更不像機器人。

Lesson 04

半自動 > 全自動（對高成本、低頻的操作）

全自動化不是終點，正確的操作才是。 送錯不能撤回的場景，保留 1–2 個人工 checkpoint 反而降低整體風險。填表機械性交給機器，最終送出與 Git push 保留給人。

Lesson 05

存證 > 快速

每次送件留 4 張截圖 + HTML + YAML + PDF，單筆約 1–2 MB。日後稽核（VCCI 回報問題、PM 想知道當時填了什麼）就有得追。Git + GitHub private repo 是「版本控管 + 異地備份 + 追溯性」一次解決的最便宜方案。

Lesson 06

生命週期 > 單次操作

第一版只做送件，結果送完後不知道哪款已 Accept、哪款還在審、什麼時候能賣。補上「狀態同步循環」才完整。對任何「送出 ≠ 生效」的申報系統，永遠要再問：送完後怎麼知道狀態？狀態改變了我要不要做什麼？有沒有辦法把這個週期也自動化？把「單次操作」擴大到「生命週期追蹤」，工具的價值會翻倍。

Lesson 07

Piggyback — 零成本的附加動作

需要的資料「正好在你已經打開的頁面上」時，順手做是最划算的：送件流程本來就會去 Top 頁抓 Report No.，這張頁面就有所有 pending 送件的狀態。多一個 parse 呼叫，零額外 HTTP request，就等於每次送新品都順便幫所有舊品做一次狀態同步。設計任何 workflow 時都可以問一次：「我這趟還在這裡，還能順手抓什麼？」

AI 時代

與 AI Agent 協作的新工作模式

整套工具（送件 + 狀態同步 + dashboard）從無到有建置約花 3 個工作日，全程用 Claude Code 對話式開發。觀察到幾個值得推廣的做法：

⚡
讓 AI 直接執行、不要只給建議 以前用 ChatGPT 要複製貼上程式碼。Claude Code 可以直接執行 uv run、git commit、開瀏覽器測試。一個 bug 從發現到修好通常 < 2 分鐘，迭代速度快 10x。
📦
用 Skill 封裝重複性任務 vcci-submit 和 vcci-status-sync 兩個 skill 寫好一次，之後要送件只要說「幫我跑 VCCI」、要查狀態只要說「查 VCCI 狀態」。等於把「資深員工的操作知識」文件化 + 可執行化。
🧩
一個專案、多個 skill（職責分離） 送件跟查狀態是不同使用情境，混成一個 skill 會讓 Claude 誤判。拆成兩個 skill，共用同一個 Python 專案的底層工具（兩邊都用 status_sync.py），各自的 SKILL.md 寫清楚觸發條件與邊界。
🔗
Private repo + Vercel 作為協作介面 跟 Claude 的對話不會自動變成團隊資產。把完成的工具放 GitHub private repo、每次送件 / sync 自動 commit + push、生成的 Dashboard 經 Vercel 上線、密碼分享給團隊 → 變成團隊隨時可以看、可以接手的實作成果。
🛡️
人工 checkpoint 就像 code review 半自動的「最終確認」步驟，等同於真人對 AI 做的 code review。保留這個步驟讓我們對工具有信心 — 知道最壞情況人還能擋下來。

核心能力可以直接重用：PDF 萃取模組、Playwright persistent context 架構、Excel 追蹤表 + 靜態 Dashboard、狀態同步循環 + snapshot 機制、Git 歸檔 + skill 編排。
