VCCI 日本認證登錄自動化

痛點、架構與可複用的學習
2026-04-19
TL;DR

20–30 分鐘的申報 → 5 分鐘搞定

EnGenius Networks Japan 每款新產品上市前都要在 VCCI 官網登錄。我們把 PDF 測試報告 → 自動填表 → 送出 → 歸檔的整條流程做成半自動工具。

過程中碰到很多官方 CGI 網站的反自動化機制 — 合成事件被擋、AJAX 需要真實 blur、雙語 PDF 錯位。最終靠 Playwright 的 page.mouse.click()（產生 OS 級真實滑鼠事件）+ Cookie 持久化 profile 繞過所有障礙。

整套架構可複用到 FCC / CE / KC 等其他國家認證送件，甚至政府申報、供應商入口等「無 API 的 UI 自動化」場景。

背景

為什麼要做這件事

EnGenius Networks Japan K.K. 在日本販售網通商品，每一款新產品上市前必須在 VCCI（電波障害自主規制協議会）官網登錄產品符合性報告。流程雖然不複雜，但：

  • 重複性高（每月 1–2 款）
  • 欄位細碎（13 個必填欄位 + AJAX 聯動）
  • 送錯難撤回（需要 VCCI 介入）

是典型「低頻、高單筆成本、易出錯」的工作，非常適合 AI + 半自動化處理。

痛點

自動化碰到的技術障礙

政府/法規類網站有很多反自動化機制，不像一般 SaaS 那樣 API-first。以下是我們實戰發現的主要障礙：

🔒 痛點 1：多層驗證（Basic Auth + Form Login）

官方網站常有兩層以上的驗證：第一層是瀏覽器級的 HTTP Basic Auth 彈窗，第二層才是網站內的 Email + Password 登入表單。Playwright 處理第一層靠 http_credentials 參數，但第二層才是真正難搞的地方。

🚫 痛點 2：合成事件不被信任（最大障礙）

VCCI 的 CGI 會靜默拒絕所有「非 OS 級」的 click 事件。以下四種方式測試結果：

Click 方式 事件類型 VCCI 反應
element.click()（JS 呼叫） 合成事件 靜默忽略
page.locator(...).click()（Playwright） 合成事件 靜默忽略
page.keyboard.press('Enter') 合成事件 不穩定
page.mouse.click(x, y)（OS 座標點擊） 真實事件 ✅ 接受

關鍵發現：page.mouse.click() 直接移動指針到座標、按下、放開，產生的是作業系統 input event，VCCI 會當成真人點擊。這是 Playwright 的隱藏武器。

🎯 痛點 3：AJAX 聯動需要真實 blur 事件

「Test Lab Member No.」欄位填完後應該 AJAX 自動帶出公司名。但 page.fill()（直接 set value）不會觸發 AJAX，因為沒發 blur。解法是模擬真實打字 + Tab：

Python / Playwright
field.click() field.press_sequentially(value, delay=30) field.press("Tab") # 觸發 blur → 啟動 AJAX page.wait_for_function("...CompanyName 有值")
📑 痛點 4：雙語 PDF 解析錯位

TÜV Rheinland 測試報告是德英雙語並列，兩個語言的 label 在文字流裡交替。用英文 label 當 regex anchor 會抓到下一個欄位的值（因為德文值先出現）。解法：改用德文 label 當 anchor，值就在同一行。

🔀 痛點 5：CGI 表單 — 一個 URL 多個狀態

VCCI 是老式 CGI 系統，所有頁面都是同一個 URL（Manager.pl），伺服器根據 POST 資料決定回哪個頁面。不能靠 URL 變化判斷進度，要 poll 下一個預期元素是否出現：

偵測頁面轉換
page.wait_for_selector( "input[type='submit'][value='Report']", timeout=30_000 )
🤖 痛點 6：Playwright 指紋被識別

VCCI 會檢測 Playwright 預設的 navigator.webdriver 等特徵拒絕登入。解法組合：真實 User-Agent + launch_persistent_context 真實 profile + 讓使用者手動登入一次，之後 cookie 持久化，繞過登入環節的 bot 檢測。

可複用學習

給其他自動化專案的 5 條教訓

Lesson 01

先確認「真人怎麼做」，再想「怎麼自動化」

剛開始以為 form login 可以直接 page.fill() + page.click()。搞了兩小時才發現 VCCI 根本不接受合成 click。一定要先用 Chrome DevTools 或 Claude in Chrome 手動走一次流程、錄下真實 HTTP request 與事件，再設計自動化，才能避開誤入歧途。

Lesson 02

page.mouse.click() 是 Playwright 的隱藏武器

大多數教學只講 locator.click()。但當遇到「合成事件被擋」的老舊 CGI 時，page.mouse.click(x, y) 才是真正產生 OS 級事件的方法。步驟是：scroll_into_view → bounding_box 算座標 → mouse.move → mouse.down → mouse.up。

Lesson 03

Cookie 持久化比「每次登入」更穩

launch_persistent_context + 固定 user_data_dir 讓 cookie 跨執行保留。好處：只要手動登入一次（繞過登入環節的 bot 偵測）→ 後續 run 全自動 → Profile 有真實瀏覽指紋，比 fresh context 更不像機器人。

Lesson 04

半自動 > 全自動（對高成本、低頻的操作）

全自動化不是終點，正確的操作才是。 送錯不能撤回的場景，保留 1–2 個人工 checkpoint 反而降低整體風險。填表機械性交給機器，最終送出與 Git push 保留給人。

Lesson 05

存證 > 快速

每次送件留 4 張截圖 + HTML + YAML + PDF，單筆約 1–2 MB。日後稽核（VCCI 回報問題、PM 想知道當時填了什麼）就有得追。Git + GitHub private repo 是「版本控管 + 異地備份 + 追溯性」一次解決的最便宜方案。

AI 時代

與 AI Agent 協作的新工作模式

整個工具從無到有建置約花 2 個工作日，全程用 Claude Code 對話式開發。觀察到幾個值得推廣的做法：

⚡
讓 AI 直接執行、不要只給建議 以前用 ChatGPT 要複製貼上程式碼。Claude Code 可以直接執行 uv run、git commit、開瀏覽器測試。一個 bug 從發現到修好通常 < 2 分鐘，迭代速度快 10x。
📦
用 Skill 封裝重複性任務 ~/.claude/skills/vcci-submit/SKILL.md 寫好一次，之後每次要送件只要說「幫我跑 VCCI 登錄」就會執行完整 pipeline。等於把「資深員工的操作知識」文件化 + 可執行化。
🔗
Private repo 作為協作介面 跟 Claude 的對話不會自動變成團隊資產。把完成的工具放 GitHub private repo、每次送件自動 commit + push → 變成團隊隨時可以看、可以接手的實作成果。
🛡️
人工 checkpoint 就像 code review 半自動的「最終確認」步驟，等同於真人對 AI 做的 code review。保留這個步驟讓我們對工具有信心 — 知道最壞情況人還能擋下來。
