小工具架構選型指南

資料來源、框架、部署平台與資料庫的選擇邏輯
2026-04-01
TL;DR

兩個問題決定一切

① API Key 需要保密嗎？ → 是的話就需要後端（Next.js），否則純前端就夠。

② 資料需要跨用戶共享或持久儲存嗎？ → 是的話才需要資料庫（Supabase），否則不需要。

大多數內部小工具，這兩個問題的答案都是「否」，靜態網頁或純前端 SPA 完全夠用。

三種常見情境

框架

三種框架的定位

最輕量

靜態 HTML

  • 單一 .html 檔案
  • 零 build step
  • 資料內嵌在 JS
  • 單人維護最省力
純前端

Vite + React SPA

  • 元件化，邏輯清楚
  • 需要 build 打包
  • 適合互動複雜工具
  • 多人協作更好維護
需要後端

Next.js

  • API Routes 藏住 Key
  • Vercel 原生整合
  • 支援 SSR / 動態路由
  • 適合需要保密憑證
💡
靜態 HTML vs Vite + React SPA 兩者概念相同（純前端、零後端），差別在於有無框架帶來的元件化與自動 re-render。靜態 HTML 是「手工 SPA」，自己實作了狀態管理和畫面更新。

資料庫

什麼時候需要 Supabase？

有「寫入」或「多用戶差異化」的需求才需要 DB。只是「讀取 + 顯示」通常不需要。

需求需要 DB？
只有自己或固定人員看，資料自己管理❌ 不需要
抓 API 資料即時顯示，不需留存❌ 不需要
「給同事看」但資料由你統一管理❌ 不需要
多個用戶各自有帳號和資料✅ 需要
需要記錄歷史、累積用戶行為✅ 需要
用戶可以寫入（留言、表單送出）✅ 需要
不同用戶看到不同內容✅ 需要
⚠️
常見誤解 「給多人看」不等於需要資料庫。「多人看」只是讀取，只要所有人看到同一份資料，靜態網頁完全夠用。需要 DB 的是「多人寫入」或「每個人看到不同資料」。
部署

部署平台選擇

平台最適合特點
Cloudflare Pages 靜態 HTML、Vite SPA 免費、全球 CDN、無流量上限，純靜態首選
Vercel Next.js、Vite SPA Next.js 原生支援、Serverless Function 整合簡單
GitHub Pages 靜態 HTML 最簡單，功能最少
💡
Vercel 的優勢是 Next.js 生態系。純靜態網頁不需要搬到 Vercel，Cloudflare Pages 無流量限制且速度更快。
架構模式

Git-based CMS

當靜態網頁需要「可更新的資料」時，透過 GitHub API 寫入 commit 觸發自動部署，就是 Git-based CMS 模式。

Admin UI 操作資料 ↓ GitHub API（Personal Access Token 認證） ↓ push commit 到 repo ↓ Cloudflare Pages 偵測變更自動部署 ↓ 網站更新（約 30 秒）

優缺點

優點限制
零後端成本 PAT 若外洩有風險（存 sessionStorage 可降低風險）
每次更新都有 git commit，天然版本控制 更新速度取決於部署時間（約 30 秒）
可一鍵回滾資料 不適合高頻寫入（例如每分鐘更新）

市場上的同類產品：Netlify CMS、Tina CMS、Forestry

最佳實踐

資料獨立成 data.js

當靜態 HTML 隨著資料累積開始變得難以瀏覽時，最小改動是將資料拆分到獨立的 data.js，概念完全相同但關注點分離。

現在（資料混在一起）
// index.html <script> const SEED_DATA = [ /* 所有資料 */ ]; // 幾千行 UI 邏輯... </script>
拆分後（更乾淨）
// data.js（只放資料） const SEED_DATA = [ /* 所有資料 */ ]; // index.html 引入 <script src="data.js"></script> <script> // UI 邏輯，不動 </script>
✅
不需要換任何框架或工具。改動極小，但讓「更新資料」和「修改邏輯」完全分開，Git diff 更清楚。
FAQ

常見問題

Q給同事看是不是就需要資料庫？
不需要。「多人看」只是讀取動作，只要所有人看到同一份資料，靜態網頁完全夠用。需要 DB 的是「多人寫入」或「每個人看到不同資料」的情境。
Q一定要用 React/Vue 才算正規？
不是。框架是為了解決複雜度與團隊協作問題。單人維護、互動簡單的工具，手寫 JS 反而維護成本更低，不是「將就」，是正確選擇。
QVercel 比 Cloudflare Pages 更好？
不是絕對的。Vercel 的優勢是 Next.js 生態系。純靜態網頁用 Cloudflare Pages 更合適（無流量限制、全球 CDN、速度也快）。
Q資料內嵌在 JS 是標準做法嗎？
不算標準做法，但對「單人維護、資料量小、單一用途」的工具來說是最合適的做法。業界標準是為了解決團隊協作與大量資料的問題，在不符合這些前提時套用反而是過度設計。
